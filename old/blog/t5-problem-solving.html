Problem Solving

Tell your non-tech friend a story about a time you:

Got blocked on a simple problem

This week we had to print out lines of hash symbols, increasing in length as you go down the page, into a sort of triangle shape. This was something similar to stuff I had done before, but for some reason it wasn't printing out a neat triangle, instead increasing by two instead of three (and on another part, adding whole extra lines of hashes before the wanted results). Because the function was relatively simple with only a few steps and a couple variables, and I was confident in the steps that were happening, I knew there was some small, particular issue that was happening to one of my variables.

To figure out what was happening I put console.log()s in various parts of my code so I could see what was happenig at each step. While this was a small error that had caused this, the result was rather annoying and working it out wasn't exactly satisfying as it wasn't some major issue that I fixed. It was nice to be able to work it out and move on to more fun things. When I have this problem again It's going to be much easier to work out where the issue is.

Solved a problem in an elegant way

I managed to find a way to simplify thee code for creating a chessboard in a similar exercise. Instead of having a great number of loops and if statements, I have two loops and an if. I took a moment to go through a number of the possibilities and results and found a simple mathematical statement to work out the solution rather than checking each of the values individually (requiring a lot more code). That extra moment to think it through made the coding a lot simpler. This was immensely satisfying.

Reflect on how confident you feel using the problem solving techniques and process:

Most of these I am very comfortable with (unless otherwise noted) and tend to use them for different stages of coding.

Pseudocode

Very useful for working out a general plan for function structure.

Trying something

I tend to do this a lot. Particularly when doing simple parts of the code that I generally know how it works but maybe not the specifics. That generally helps me avoid causing terrible errors or infinite loops when trying things.

Rubber ducky method<

I find this great when you're trying to write a rather complex function. Being able to describe the steps and what should be happening without worrying about the specific code you should be using is very handy when the structure itself is confusing.

Reading error messages

Even if you don't exactly know what the error message is referencing, error messages are great for telling you what sort of area you should be looking at, what the problem is related to.

Console.logging

A good starting point when something should work, but it just doesn't seem to. It lets you look at each of the individual steps that is happening, and you can check that against what you expect should be happening

Googling

I find this very useful when I have a specific problem that I do not know what to do to solve it. Finding documentation on the code or seeing how other people dealt with the problem is a great starting point.

Asking your peers for help<

I haven't yet had to do this on slack, I've had a couple of conversations where we've talked about general concepts which is great for understanding things, but no specific issues I have had. I don't tend to use this for specific problems, but sometimes it's an issue everyone is having and then it's good for everyone!

Asking coaches for help

Again, I haven't had any terrible issues yet, but times when I just cannot work out how to deal with a problem I know having someone there who I can go to when all else fails will be great.

Improving your process with reflection

This isn't something I have been very good at in the past. But these past few weeks I have been trying to embrace this method and have noticed a couple times when I have actually done something in a slightly different fashion after reflecting back on it in previous weeks. So that seems promising :)
